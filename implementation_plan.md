# ZenClean (禅清) - 现代 C 盘深度清理工具终极实施方案

本项目参考 `One-click-cleaning-of-C-drive` 的扫描广度，通过 Flet 构建现代化 UI，接入 AI 智能分析引擎，并对接 `hw-license-center` 卡密授权体系实现商业化闭环。

## ⚡ 性能与并发架构

*   **IPC 进程隔离机制**：
    由于 Python 的 GIL（全局解释器锁）特性，高 CPU/高 IO 的深度扫描会严重拖垮主线程。扫描引擎将被彻底剥离为独立的子进程（使用 `multiprocessing` 或隐式子 exe），主进程纯粹负责 Flet 的 UI 响应与通信交互（利用 `Queue` / `Pipe`）。

## 🛡️ 安全、恢复与容错闭环 (核心命门)

清扫工具一旦误删就是灭顶之灾，同时又要保证 **“用户必须第一眼看到 C 盘空间激增的爽快感”**。我们搭建以下软硬隔离防线：

1. **绝对不可达白名单 (Hardcoded Safeguards)**：
   在系统底层硬编码锁定 `C:\Windows\System32`、`Windows Defender` 等致命区域。哪怕正则走火或 **AI 返回结论判定“可清理”**，都会被这层防御直接丢弃。
2. **Junction 环路防死锁检测**：
   扫描阶段必须校验 `os.path.islink()` 防范无限循环递归。在向非 C 盘创建 Junction 前，严苛判断目标目录不是源目录的子集。
3. **“双重分诊”粉碎法 (Triage) —— 兼顾安全与空间释放**：
   - **绝对安全项 (LOW 级)**：如 `%TEMP%`、浏览历史等毫无争议的垃圾。直接调用系统 API **物理粉碎**，瞬间释放极限 C 盘空间，满足核心爽感。
   - **争议/未知项 (MEDIUM/HIGH 级)**：如开发者缓存或 AI 建议的冗余环境。使用 **`send2trash` 仅移动至回收站**，保留“最后一颗后悔药”。
4. **终极清理仪式感与免责确认**：
   在完成上述清理后，UI 提供红色的“物理清空回收站争议项”按钮。点击前强制弹窗免责警告（“若系统异常请勿清空回收站！”），将误删的最终确认权和风险转移给用户。
5. **创建系统还原点 (SRSetRestorePoint)**：
   **终极杀手锏 (第三阶段实现)**。执行 VIP 级别的开发环境极客粉碎或跨盘 Junction 连接前，强制通过 `ctypes` 调用 Win32 API 创建还原点，保障即便注册表被污染也能系统级光速 Time Travel (时间穿梭)。
6. **物理自动备份保留 (.temp_bak)**：对于极高危的文件替换，打包至临时安全屋以便 UI 一键回滚。

## 🧠 AI 引擎成本与隐私清洗

1. **敏感数据的脱敏隔离 (Sanitization)**：
   本地代理拦截向云端发送的路径字符串前，执行抹除脱敏操作。譬如强制将 `C:\Users\Admin\Desktop\私人合同.pdf` 剥除身份层级，甚至替换为 `%USERNAME%` 虚拟占位符发送，避免深潜数据外泄。
2. **本地 AST Hash 命中缓存池**：
   将如 `node_modules` 这类泛化且千篇一律的前端特征树作 MD5 / 指纹缓存。只要 AI 判定这套树结构安全，后续即便是新建立的项目也直接命中本地沙箱放行，**不消耗 1 分钱的 API Token**。
3. **Token 限流阈值 (Batch Control)**：
   积压型打包发送，一次汇编 50+ 个未知扩展名路径给云端 API（调用中转智谱 **GLM-4-Flash**，基于 `openai` 官方库）打包判定。并在个人配置文件中设置日频请求配额。
