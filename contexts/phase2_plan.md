# 阶段二 (护城河防御与热修复版) 实施方案 v1.0

回顾第一阶段（MVP版），我们的开发进度实际上**大幅超前**。原本规划在后续阶段的诸多硬核特性（Flet 列表万级虚拟化渲染、NTP 时间防回退鉴权、`py-machineid` 硬件机器码绑卡，乃至于云端 AI SSE 流式大模型网关链路），在这几天高强度的冲刺中均已**提前落地验证**。

因此，**真实的第二阶段**，我们的火力将更加聚焦于构建 ZenClean 作为“商业级系统优化软件”的两大终极生命线：**绝对安全的后悔药体系** 与 **免发版的云端热修复体系**。

---

## 1. 核心任务定义

第二阶段的开发可以划分为以下三大核心战役（Task Clusters）：

### 战役一：系统级安全兜底机制（后悔药）
虽然第一阶段我们通过 `send2trash` 实现了风险文件移入回收站的中转保护，但对于极客用户和重度清理场景，我们需要比回收站更彻底的系统级容灾。

*   **Task 1.1: 创建系统还原点 (SRSetRestorePoint)**
    *   **目标**：在用户执行含 `HIGH` 高危风险文件的全量清理前，强制插入系统级快照节点。
    *   **实现**：调用底层的 `ctypes.windll.srclient.SRSetRestorePointW`。
    *   **UI 联动**：在清理前置位弹出转圈动画提示：“正在为您创建系统级防毁快照...”。一旦清理导致 Windows 注册表污染或系统级软件瘫痪，用户可通过 Windows 自带系统还原时空穿梭。

*   **Task 1.2: 离线物理沙箱备份 (.temp_bak 暂存区)**
    *   **目标**：针对部分无法使用快照的孤岛文件执行硬拦截移存。
    *   **实现**：在应用层实现文件移动至 `%APPDATA%\ZenClean\Quarantine` 隔离区，并维护一个 JSON 索引清单。
    *   **UI 联动**：开发一个新的独立视图（如“隔离恢复舱”页面），允许用户点击单行文件进行原路 `shutil.move` 还原。

### 战役二：深度清理进阶能力 (Deep Clean)
传统清理软件往往将所有垃圾打包丢给用户，而 ZenClean 坚持“克制优先”。但在第二阶段，我们将为高级用户开放“核弹级”开关。

*   **Task 2.1: 独立“深度清理”视图页面**
    *   **目标**：在左侧导航栏新增 `Deep Clean` 面板，与基础的“智能清扫”严格物理隔离。
    *   **策略**：在此页面解除基础版规则限制，将 `.NET 预编译缓存`、`Windows 组件装配集`、`浏览器深层 Cookies/历史` 等底层数据暴露给极客用户。
    *   **UI 联动**：首次进入时需弹出高危免责确认框，强化极客属性与风险认知。

*   **Task 2.2: 系统底层瘦身 (System Optimize)**
    *   **休眠文件管理**：检测 `hiberfil.sys` 状态，支持一键切换（执行 `powercfg -h off/on`），平均释放 8~32 GB。
    *   **虚拟内存引导**：展示 `pagefile.sys` 大小，提供快捷入口直达系统性能设置视图。

*   **Task 2.3: 僵尸项目猎手 (Zombie Dev)**
    *   **目标**：扫描全盘（或 C 盘）`node_modules` 目录，计算大小及最后修改时间。
    *   **意义**：清理陈旧、不再维护的临时开发项目，通常每个项目可腾出 500MB+ 空间。


### 战役三：知识库与版本热更体系 (Hot Update)
AI 虽然聪明，但响应速度和成本永远比不上本地正则匹配。为了防止出现“某款热门软件新版改了缓存结构导致被误删/漏删”，我们要让本地规则库具有云端同步的能力。

*   **Task 3.1: `file_kb.json` 规则树热更 (OTA)**
    *   **目标**：剥离写死的规则，启动时静默由 `hw-license-center` (或 GitHub Raw/CDN) 拉取最新的 `.json`，校验 SHA256 无篡改后更新至本地缓存。
    *   **实现**：后台异步线程，基于时间戳检测（`Etag` 或 `If-Modified-Since`）。
    *   **UI 联动**：如果发生了规则更新，底部状态栏短暂弹出 Toast：“本地规则引擎已更新至 v1.x 版”。

*   **Task 3.2: EXE 自升级通道 (Auto-Updater)**
    *   **目标**：基于此前已写好的 GitHub Release 检查机制，补齐“下载 ➜ 重命名替换 ➜ 重启进程”的闭环，让用户在端内点击即可跨版本升级，无需跳网页。

### 战役四：安全提权与打包分发 (Packaging & Evasion)
为了真正成为可以在任意用户电脑上运行的桌面端软件，需解决权限和反破解。

*   **Task 4.1: UAC 自动提权 (Admin Privileges)**
    *   **目标**：清理底层垃圾和生成系统还原点必须具备 Windows 管理员权限。
    *   **实现**：在 `main.py` 入口点探测 `ctypes.windll.shell32.IsUserAnAdmin()`。若为假，则通过 `ShellExecute` 结合 `runas` 动词发起 UAC 白盾弹窗要求重载进程。

*   **Task 3.2: 反脱壳与密钥混淆 (PyArmor / Cython)**
    *   **目标**：保护 `core/auth.py` 与 `ai/cloud_engine.py` 等商业逻辑核心引擎，防止逆向破解“修改本地 JWT 验签代码白嫖 VIP”。
    *   **实现**：在发布流水线中引入 PyArmor/Cython 进行符号抹除及混淆加密。

*   **Task 3.3: 单体免杀打包规范化**
    *   **目标**：收敛 PyInstaller `.spec` 脚本，将 Python 环境、`flet` 的 Webview2 依赖及所有的静态 `assets/` 资源压为最小化单一 `.exe`。
    *   **挑战**：排查 `multiprocessing` 在打包单体程序（`--onefile`）模式下可能引发的无尽自我 Spawn (炸服) 循环。

*   **Task 4.4: 封装安装程序 (Inno Setup 迁移)**
    *   **二期任务**：在发布时捆绑微软官方离线安装包，通过 Inno Setup 制作专业的 Setup 安装程序，在启动前静默检测并全自动补齐环境。

*   **Task 4.5: 申请商用数字签名 (Ev/Standard Sign)**
    *   **目标**：消除“未知发布者”警告，提升下载安全性，减少杀软误报。
    *   **计划**：在三期或正式版发布前完成证书采购与签名流水线对接。

### 战役五：法律风险规避与数据合规 (Compliance)
本部分动作已在进入第二阶段前夕**提前实施完毕**，作为构筑 ZenClean 商业安全的生命底线。

*   **Task 5.1: 明示同意免责机制 (EULA & Privacy)**
    *   **目标**：通过强制用户界面拦截，规避极限误删操作引发的直接经济索赔诉讼。
    *   **已实现**：编写了 `TERMS_OF_SERVICE.md` 和 `PRIVACY_POLICY.md`。并在 `auth_view.py` 的激活入口插入强制复选框拦截逻辑（未勾选则红色报错无法进入主功能）。
    
*   **Task 4.2: 双路数据隔离防线 (Isolation)**
    *   **目标**：防止个人计算机上的隐私文件名（如敏感企微缓存、含机密字眼的合同路径）被直接上传至第三方大模型造成泄露。
    *   **已实现**：
        1.  **AI 脱敏器 (`sanitize_path`)**：在 `local_engine.py` 层将路径身份层级强制替换为 `%USERNAME%` 等沙箱占位符。
        2.  **软删除中转 (`send2trash`)**：除系统 Temp 外一切未知文件皆执行回收站转移。确保清理动作存在反悔撤销窗口，将最终数据灭失权转移给用户自行清空动作。

## 2. 第二阶段实施路线图 (Timeline)

- **Week 1 (进阶清洗与兜底)**：实现深度清理 (Deep Clean) 独立视图引擎；攻坚 `SRSetRestorePoint` 系统级防毁快照。
- **Week 2 (核心热更)**：搭建 `file_kb.json` 热同步协议与云端 CDN 资源对齐。
- **Week 3 (打包发布)**：针对 PyInstaller 陷阱排雷；进行核心模块 Cython 编译测试；使用 Inno Setup 封装包含一键安装环境的最终版本。

## 3. 下一步行动建议

在正式进入上述极其硬核的代码编写前，我建议我们以 **Task 3.1 (UAC 自动提权)** 或者 **Task 1.1 (系统还原点创建)** 作为第二阶段第一炮！这两个功能直接关乎到我们软件执行时获取底层文件访问的最高权限。
