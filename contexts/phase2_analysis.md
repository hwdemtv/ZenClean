# ZenClean 第二阶段可行性分析与稳妥实施方案 (v1.0)

基于对当前项目结构（`src/`）以及核心模块（`core/auth.py`, `core/cleaner.py`, `core/updater.py`, `main.py`）的现状剖析，以下是对 `contexts/phase2_plan.md` 中各项任务的可行性评估与稳健实现建议。

---

## 战役一：系统级安全兜底机制（后悔药）

### 1.1 创建系统还原点 (SRSetRestorePoint)
- **现状评估**：未实现。
- **可行性分析**：**[高可行，但有强依赖]**
  - Windows 提供了 `ctypes.windll.srclient.SRSetRestorePointW` 原生 API 支持。
  - **最大卡点**：创建还原点必须处于 **管理员权限 (UAC elevated)**。当前 `main.py` 启动时未请求提权。如果不提权直接调用，会遭遇 `OSError: [WinError 5] 拒绝访问` 或静默失败。
- **稳妥实施方案**：
  1. **前置 UAC 提权**：在执行任何底层快照前，封装 `ctypes.windll.shell32.IsUserAnAdmin()` 进行查验。
  2. **异步非阻塞 (高级工程优化)**：`SRSetRestorePointW` 是阻塞调用。由于系统 VSS 创建快照可能耗时数秒到十几秒，必须放入独立的后拉线程（如 `threading.Thread`），并通过 Flet 弹出不可关闭的加载提示防止 UI 假死。
  3. **强制 15 秒超时装饰器机制**：利用 Python `Thread.join(timeout=15)` 封装 `@async_with_timeout` 装饰器，超过 15 秒强制放行，绝不因为兜底快照系统卡顿甚至服务损坏而卡死核心清理进程。

### 1.2 离线物理沙箱备份 (.temp_bak 暂存区) [跨盘符优化版方案]
- **现状评估**：未实现。目前 `cleaner.py` 依赖 `send2trash`（移入回收站）。
- **可行性分析**：**[技术可行，但将深刻改变产品形态（变重）]**
  - **优势**：真正能把 C 盘的空间释放出来。当扫描出 5GB 的争议大文件时，如果目标沙箱在 D 盘，那么迁移后 C 盘确实会多出 5GB 空间，这实现了“清理”的根本目的。
  - **高危痛点（跨盘 IO 瓶颈）**：
    - 在 Windows 下，同盘 `shutil.move` 只是修改逻辑地址（瞬间完成）；而**跨盘 `shutil.move` 实际上是完整的“复制 + 耗时读取 + 删除原文件”的物理拷贝过程**。
    - 如果用户的硬盘是机械硬盘（HDD）或文件极其细碎（如几万个 Node.js `node_modules` 小文件），这 5GB 的体积可能导致长达 **5~15 分钟** 的文件传输阻塞。
    - 用户原本期待“点击清理 ➜ 3秒动画 ➜ 释放完毕”的畅快感，将会被漫长的进度条无情摧毁，变成一个重度的“数据迁移工具”。
- **稳妥实施方案（如必须执行）**：
  鉴于清理工具强调的是“快”与“爽感”，如果我们要引入跨盘 `.temp_bak`：
  1. **不能默认开启**：绝不能把它混在核心的“一键体检”流水线里。此举会拖垮整个工具的用户体验。
  2. **定位为独立的 VIP 专属工具 ——「高危地带保险箱迁移」**：
     专门拿出一个 UI 面板（类似于现有的“系统文件夹无损搬家”），让极客用户手动勾选那些高风险的开发者目录，并在弹窗中明确警告：“**此保护性迁移操作涉及跨盘符物理拷贝，预计耗时数分钟，期间请勿关闭程序。**”
  3. **底层实现约束**：必须确保选中的目标盘符（如 D: 或 E:）有足够承载此次备份的剩余空间，且需在 `cleaner.py` 中引入专用的 `shutil.copytree` 进度回调以更新 Flet 界面进度条，防避 UI 假死。
  4. **常规日常清理维持原判**：对于绝大多数垃圾文件，依旧坚守原生的 `send2trash` 回收站逻辑。

## 战役二：知识库与版本热更体系 (Hot Update)

### 2.1 `file_kb.json` 规则树热更 (OTA)
- **现状评估**：未实现。目前使用本地静态 JSON。
- **可行性分析**：**[极高可行]**
  - 当前架构通过 `json.load()` 将规则加载到内存，天然解耦。
- **稳妥实施方案**：
  1. 启动时（或闲时）派生后台线程，请求远端 `kb.json`。
  2. **必须引入防投毒校验**：拉取内容前必须校验 SHA256 签名（通过公钥或简单的后端盐值）。若被中间人篡改（例如下发规则将 `C:\Windows` 标记为 `LOW` 级直接粉碎），后果不堪设想。
  3. **增量更新机制 (高级工程优化)**：不要每次 ETag 变动全量拉取几 MB 大小的规则文件。设计类似 `/api/v1/rules/diff?from=v1.0` 的差异接口，仅下发如 `+ruleXY`, `-ruleZ` 的极简补丁（Patch），这能给服务端极其可观的流量带宽压降。

### 2.2 EXE 更新提示通道 (跳过端内自升级)
- **现状评估**：`updater.py` 仅实现了请求 GitHub API 对比版本号并回调通知。
- **可行性分析**：**[端内覆盖热更门槛极高，容易死锁，体验糟糕]**
  - Windows 系统严格**锁定正在运行的 .exe 文件**，不允许本体在运行态被自身覆盖替换。
- **稳妥实施方案**：
  - **听取极简建议：只提供更新链接**。
  - 维持现有基础策略：检测到新版本后，在界面顶部闪过一条 Snackbar，或者在“关于”页面高亮提示。
  - 用户点击更新按钮时，直接调用 `flet.page.launch_url()` 拉起系统默认浏览器，跳转到 GitHub Release 页面或官网下载页。这极大减少了开发风险，且符合此类清理软件的惯性。

## 战役三：安全提权与打包分发 (Packaging & Evasion)

### 3.1 UAC 自动提权 (Admin Privileges)
- **现状评估**：尚未实现。
- **可行性分析**：**[必须首优实现]**
  - 清理深层系统路径（如 `Windows/Temp` 某些死角）及创建系统还原点，权限不可或缺。
- **稳妥实施方案 (按需与记忆提权策略)**：
  - **不要一开机就弹窗要 UAC 权限**，这极容易被 360/火绒判定为开机木马并吓跑用户。
  - **按需索要**：在用户点击“深度扫描”等需要特权的按钮时，再调用 `ShellExecuteW(..., "runas")` 重启自身。
  - **用户选择状态记忆**：将用户同意提权存入 `%APPDATA%\ZenClean\config.json` 当中（`user_declined_admin: false`），下次启动便自动提权，极大提升日常操作顺畅感。

### 3.2 反脱壳与密钥混淆 (PyArmor / Cython)
- **现状评估**：纯 Python 明文运行。
- **可行性分析**：**[高可行且必要]**
- **稳妥实施方案**：
  - **切忌全量混淆**：不要用 PyArmor 混淆 Flet 的 UI 代码（会导致难以名状的重绘卡顿和打包后巨大增量）。
  - **精准打击**：仅对 `core/auth.py`（加密鉴权验证逻辑）和 `config/settings.py`（密钥/域名硬编码）使用 Cython 编译为 `.pyd`（C 扩展模块）即可。这足以抵御 99% 的脚本小子的非法摘除联网校验的白嫖行为。

### 3.3 单体免杀打包规范化
- **现状评估**：目前由开发者手动运行脚本，但 `main.py` 底端已预留 `multiprocessing.freeze_support()`。
- **可行性分析**：**[高危雷区]**
  - PyInstaller `--onefile` 模式下，结合 `multiprocessing`，如果在 Windows 上漏掉 `freeze_support()`，会无限衍生子进程直至内存枯竭导致蓝屏（Fork Bomb）。
- **稳妥实施方案**：
  - 确信 `main.py` 底部保护代码完好无损。
  - 使用 `--onedir` （文件夹模式）作为阶段二主流分发测试方式（避开 `--onefile` 每次启动漫长解压临时文件夹的问题），这也是为阶段三用 Inno Setup 铺路。

## 战役四：开发者权益保护与防破解体系 (Anti-Piracy & Protection)

鉴于清理软件属于极易被“脱壳、拔网线白嫖”的工具类软件，我们必须在第二/三阶段建立一套多维度的坚固防线，保护作者辛勤手搓的大模型 Prompt 和商业逻辑。

### 4.1 核心资产的物理层防御 (Cython 强加密)
- **现状与痛点**：目前全版为 `.py` 明文。一旦使用 PyInstaller 打包，黑客只需使用 `pyinstxtractor` 加上 `uncompyle6`，五分钟内就能拿到所有源码，并轻易注释掉 `auth.py` 中的 `verify_license_online` 函数返回 True。
- **坚固防御方案（选择性 Cython 编译）**：
  - **绝不全量加密**：PyArmor 这类工具对 Flet（基于内嵌 Flutter 引擎和大量异步事件）有致命影响，极易导致白屏和巨量性能损耗。
  - **外科手术式保护**：利用 `Cython` 将涉及商业命脉的**仅 3 个文件**编译为 `.pyd`（C 扩展二进制库，逆向难度等同于破解大型 C++ 游戏）：
    1. `core/auth.py`（授权校验心跳逻辑）
    2. `ai/cloud_engine.py` (请求服务端 AI 网关的 API 结构与签名)
    3. `config/settings.py` (所有的加密盐、API URL 和特服白名单)

### 4.2 API 接口防刷与云端限流 (Gateway Rate-Limiting)
- **现状与痛点**：目前的云端 AI 依靠 `hw-license-center` 中转，如果不加限制，一旦有人抓包拿到合法的 JWT Token，就能写脚本疯狂并发请求，耗尽我们的 智谱清言 余额（即“API 盗损”）。
- **坚固防御方案**：
  - **设备强绑定 (Device Fingerprinting)**：现有的 `core/auth.py` 已经通过 `get_device_id()` 获取了硬件指纹并和 Token 绑定。在向 AI 网关发起请求时，必须在 HTTP Header 中携带此指纹。服务端核对 Token Payload 中的 `device_id` 与请求头中的是否一致，不跨机使用。
  - **银行级签名防重放机制 (HMAC + Nonce)**：
    抛弃常规 MD5，请求附加 `X-Sign: HMAC-SHA256(Method+URI+Timestamp+Nonce, Secret)`。
    - 服务端验证时间戳偏差不过 60 秒。
    - 服务端将 `Nonce` 存入 Redis 并设置 60s 过期，只要拦截到用过的 Nonce 直接判定为重放攻击恶意调用并斩断。完全堵死抓包后狂发千次的白嫖通道。
  - **服务端速率断路器**：在此前设定的每日 50 次额度基线之上，增加**每分钟 5 次的滑动窗口限流**，一旦超频直接封禁该设备 IP 24 小时。

### 4.3 离线断网白嫖防御 (NTP 时光机检测)
- **现状与痛点**：用户购买了 1 个月的 VIP，在第 29 天断网运行，并通过修改 Windows 系统本地时间，无限欺骗本地 JWT 的 `exp` (过期时间)。
- **坚固防御方案**：
  - 目前 `auth.py` 中的 `_check_time_drift()` 已经初具雏形，但不够激进。
  - **演进策略**：在有网状态下，启动时**必须**校验 NTP 时间与本地时间偏差（超过 300 秒拦截）。在**真正断网**无服务端响应的状态下：
    - 在 `%APPDATA%\ZenClean\sys.dat` 隐秘写入**最后一次合法启动的本地时间戳**（需 AES 加密）。
    - 每次无网启动时，如果发现当前系统时间**早于**这个隐秘记录的时间戳，直接判定用户在“倒拔系统时钟”，当场吊销本地 JWT 缓存并退出软件。

### 4.4 反调试与反内存注入 (Anti-Debug)
- **可行性分析**：防君子不防高端黑客。
- **软风控着陆策略 (高级避坑建议)**：
  千万不要在入口扫描到 `Wireshark` 等进程名后**直接静默退出（闪退）**。这会引发正常程序员用户的大面积客诉（开着封包工具忘了关）。
  正确的做法是**只记录，不退出**：一旦 `psutil` 扫到黑名单进程，我们在后台默默产生一条黑产嫌疑记录。一旦服务端判断属于高危操作（如下发新版热规则失败、验证异常），切断关键能力（如限制进入激活、深度结算界面）。让普通技术网民继续用免费功能，但核心功能防偷取。

### 4.5 法规层面的铠甲 (EULA 与隐私免责)
- **现状与痛点**：技术防御即使再重，也可能被攻破（或者被羊毛党利用规则漏洞）；再安全的 AI 裁决（GLM-4），也有 0.01% 的几率误删用户“C 盘存放的价值百万合同”，如果不做法律切断，开发者将面临无穷的连带赔偿勒索风险。
- **坚固防御方案（法律协议隔离网）**：
  1. **《最终用户许可协议 (EULA)》强制打断**：
     - 在首次安装（第三阶段 Inno Setup）与首次激活（`auth_view`）时，**必须且只能由用户主动勾选**“已阅读并同意免责条款”。
     - **核心条款主张：技术无罪与“原封退款限度”**。明确指出产品为按现状（"AS-IS"）提供的效率工具，开发者**最大赔偿责任不能超过用户购买软件的实际所付金额**。
  2. **“最后一击”风险转移机制 (知情同意权)**：
     - 清理逻辑之所以必须把争议文件移入回收站（而不是直接粉碎），这是最强的法律盾牌。
     - 在软件 UI 上的最后一步“彻底清空回收站争议项”必须用刺眼的红色按钮，点击时必须弹出**免责确认框**（例如：“此操作完全出于您的个人判断，清空后数据灭失与 ZenClean 开发者无关”）。这在法理上完成了“过错转移”。
  3. **反向工程法律追责声明**：
     - 在代码包根目录以及 EULA 中写明：任何试图脱壳（Cython/PyArmor）、抓包或篡改心跳数据的行为（逆向工程），即视为**侵犯商业秘密（而非单纯的破解）**，开发者保留直接向法院申请网络接入端禁售及索赔的权利。这个威慑对国内很多以售卖注册机为生的淘宝工作室非常有用。

## 五、阶段二致命风险识别与硬核应对指南 (Risk Mitigation)

第二阶段不仅是“做功能”，更是“扫雷”。进入内核级别的操作必然伴随高危风险，以下是我们在编码时**必须强制执行**的应对策略：

### 🚨 1. `multiprocessing` 打包导致的死循环炸弹 (Fork Bomb)
- **风险源**：Windows 平台打包 Python `multiprocessing` 为单文件 EXE 时，若处理不当，双击启动会无限制孵化子进程，瞬间吃光内存导致电脑死机。
- **应对措施 (强制代码规范)**：
  1. **放弃单文件流**：坚决禁止使用 `PyInstaller --onefile` 进行内测与发布，改用 `--onedir` 文件夹模式，极大降低子进程环境解压冲突。
  2. **注入守护钩子**：在 `main.py` 的绝对第一行（`import`之前）必须写入 `multiprocessing.freeze_support()`；在 `PyInstaller` 的 `.spec` 文件中注入自定义的 `rthook.py`（运行时挂载 sys._MEIPASS）。

### 🚨 2. 系统还原点创建卡死 (VSS 阻塞)
- **风险源**：用户电脑的 `System Restore` 服务可能被禁用、坏块或空间不足。强制调用 Win32 API 创建快照可能会导致我们的主线程白屏假死长达数分钟。
- **应对措施 (异步降级)**：
  1. **独立线程包裹**：将 `SRSetRestorePointW` 调用完全放入独立的 `threading.Thread` 中执行。
  2. **设立绝对超时**：设定 15 秒（Timeout）断路器。超时未完成即视作“创建失败”，但**直接放行清理主流程**（记录 Warning 日志）。绝不能因为兜底快照失败而卡死主业务。
  3. **UI 镇痛剂**：在请求期间弹出一个不可点击关闭的半透明 Loading 对话框（“正在创建防毁快照...”），成功或超时后自动销毁对话框。

### 🚨 3. UAC 提权引发杀毒软件误杀
- **风险源**：如果一个未签名的普通 `.exe` 一启动就调用 `ShellExecuteW("runas")`，360/火绒大概率会弹红框报读为木马。
- **应对措施 (心智铺垫与分离)**：
  1. **启动时静默，点击时提权**：如果可以，不要一开机就要求管理员权限。允许用户以普通权限浏览界面甚至扫描普通垃圾；当点击“深度清理”或“激活”时，再弹窗说明：“为了触达深层垃圾，接下来将请求管理员 UAC 权限”。
  2. **第三阶段终极解药**：用 `Inno Setup` 做安装包时自动请求权限并在桌面快捷方式写入默认提权。

### 🚨 4. 规则热更新遭遇中间人投毒
- **风险源**：如果 `file_kb.json` 的下载被劫持，黑客下发恶意规则（如将 `C:\Windows` 设定为 `LOW` 直接删除），软件将变成毁灭性武器。
- **应对措施 (强双向校验)**：
  1. **HTTPS 强制**：只允许从 `https://` 且必须是硬编码的 GitHub/CDN 域名拉取。
  2. **JSON 数据沙箱**：在合并从云端拉取的新 JSON 前，必须走一遍 `try...except` 解析器，且强制验证其包含我们规定的必备 Key（如 `version`, `rules`）且字段类型绝对正确。
  3. **不可覆盖底线**：不论云端规则如何更新，`cleaner.py` 里的 `whitelist.py` **绝对硬编码白名单防御**等级永远最高，即使规则错判，白名单也会把它拦截下来。

## 🎯 实施结论与 ROI (性价比) 任务矩阵

综合技术可行性、商业防御价值与开发成本，我们把第二阶段（含法规防线）的任务提炼出如下**优先级推演矩阵**，这决定了我们写代码的先后顺序：
