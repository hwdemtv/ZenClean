# Phase 3 开发计划：铁壁沙箱与无损回滚 (避坑架构)

## 核心指导思想 (避坑规范)
绝不直接使用高风险的原生破坏性 API。所有深层系统修改必须遵循以下“三步走避坑机制”：
1. **隔离优于删除**：用“移动到沙箱”代替彻底粉碎，为用户保留 72 小时的“后悔药”。
2. **哨兵防线**：在执行任何针对应用（如微信、Docker）的搬家或清理前，必须在底层校验对应进程是否已彻底终止，并检查目标磁盘读写权限与可用容量。
3. **原子回滚 (Transaction)**：跨盘传输必须先 `Copy + 校验` 再删除原件，一旦断电必须允许从原盘无缝恢复。

## 一、 铁壁沙箱舱 (Quarantine Sandbox) 的建立
这是 Phase 3 的前提。没有沙箱，就不开启深度瘦身。

*   **机制**：
    *   在非系统盘（推荐 D 盘或剩余空间最大的逻辑分区）建立 `.ZenClean_Sandbox` 隐藏文件夹。
    *   在清理时，将高危垃圾文件（如疑似重要缓存）由 C 盘转移至此，并记录元数据（原路径、移动时间）。
    *   **自动销毁**：驻留超过限定阈值（如 3 或 7 天）后，由静默进程物理抹除。
*   **UI 联动**：
    *   在“禅清看板”增加一个“时光机 (恢复)”入口。
    *   用户可直观看到最近被隔离的文件列表，并一键还原。

## 二、 无损搬家解锁 (强制哨兵防护)
目前界面上的无损搬家仅为入口，在 Phase 3 将完全解封，赋予其物理引擎。

*   **处理目标**：微信 (`WeChat Files`)、部分占用高的 Electron 缓存等。
*   **操作流**：
    1.  **哨兵探活**：`psutil` 遍历进程表，发现 `WeChat.exe` 在运行，弹窗警告并高亮提示用户先关闭。
    2.  **原子性拷贝**：带进度条的复制到目标分区。
    3.  **Junction 生成**：使用 Windows `mklink /J` 在 C 盘原位置建立硬通道。
    4.  **安全抹除**：软连接确立后，清理原文件残留并标记成功。

## 三、 四色全知数据大屏
将目前的容量环形图升级，更直观透视垃圾成分。

*   **视觉拆分**：系统内核(蓝)、应用程序(紫)、可清垃圾(橙)、剩余空间(青)。
*   **交互联动**：鼠标 Hover 到对应环形片段时，下方的 `Top 5 空间探针` 随动过滤，显示对应类别里最大的文件。
*   **责任转嫁 UI**：对于 AI 置信度在 70%-90% 的文件清理，使用“滑动解锁”控件代替“点击删除”，增加用户的操作庄重感，降低产品背锅风险。
