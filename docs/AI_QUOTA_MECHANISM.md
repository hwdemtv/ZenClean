# AI 版图测绘算力 (额度) 计算规则与账单模型

在 ZenClean 架构中，客户端与“云端 AI 网关 + 智谱大模型”协同工作。为了实现极致的性能优化与成本控制，AI 分析额度的消耗遵循**“本地缓存优先 + 最小化实际请求”**的原则。

## 1. 额度消耗规则：何时扣减 1 次额度？

**严格的按次扣减原则：**
当且仅当 ZenClean 客户端遇到一处**“从历史记录中未曾见过”**的新鲜文件或目录路径时，它才会通过网络向后端的 AI 代理网关发送一次 `/api/v1/ai/chat/completions` 分析请求。
每当后端网关成功响应并下发模型判定结果后，数据库中该授权账户的今日已用额度（`used_today`）才会真实地增加 `1` 次。

**计费模型要素：**
* 客户端发起真实的网络请求。
* 请求成功通过 JWT 鉴权与流控拦截。
* 成功调用底层的大语言模型 API 并完成响应。

---

## 2. 核心降本原理：强悍的“本地零成本复用”缓存引擎

如果您一次性扫描了 10,000 个文件，**绝不意味着**会消耗 10,000 次 API 额度，因为系统内置了持久化的《大模型判据底注库》（`ai_cache.json`）。

**缓存命中流程（白嫖算力）：**
1. 客户端在准备请求 AI 之前，会首先对目标路径（如 `C:\Windows\Temp\xxx.log`）进行正则范式化或精确匹配。
2. 如果在本地缓存库中查找到该文件过去已被 AI 判定过（例如判定为 `LOW`，即安全可删）。
3. 引擎将瞬间以 **0 毫秒延时、0 网络开销**返回结果，并且**后端完全 0 扣费**。

越长时间使用 ZenClean，客户端积累的判断语料就越多，对云端算力的依赖就越小，实现了在限定额度内无上限的高频扫描能力。

---

## 3. 后端严格的“跨日重算”逻辑

我们在 Cloudflare Workers 部署的后端网关中，内置了自动的周期性快照与清零规则：

* **校验周期：** 每日的 UTC `00:00:00` 为账单日切线。
* **惰性清零机制：** 后端**不会**通过定时任务去扫表清零所有用户。而是当用户在新的一天发起**第一笔新 AI 请求**时，网关会比对本次请求的时间与该用户上一次产生消费记录的 `ai_last_reset_date`（重置日期）。
* 如果判断为跨日，后端会在处理当前请求之前，**瞬间清零 `used_today = 0`**，而后将本笔订单记为 `1`。

---

## 4. 极端场景与“无痛优雅降级”

* **额度用尽时 (HTTP 429)：**
  如果用户在一天内扫描了极其海量的“全新未知类型垃圾”，导致今日额度（例如 100 次）耗尽。此时再发出请求时，后端不响应内容，而是返回 `HTTP 429 To Many Requests`。
* **系统的兜底表现：**
  客户端底层引擎非常强壮，收到 `429` 后，**不会触发应用崩溃或假死**。引擎会启动“友好降级”模式，静默地将后续未知文件标记为 `UNKNOWN`（或保守判定为非建议删除），并在日志中记录。扫描主流程照常快速跑完。用户第二天再扫时即可继续启用精细判决功能。
